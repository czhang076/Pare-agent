## 1. What is Pare?

While SWE-agent, Aider, and OpenHands generate code, Pare is the platform that tells you whether it actually works — batch-evaluating agent output for patch validity, test coverage, and failure patterns at scale.

Pare is a **Headless-first QA & Automated Repair Engine**. It doesn't just guess if an AI-generated patch works; it evaluates it against runtime constraints, aggressively cross-validates to prevent LLM cheating, and spawns parallel repair universes if the patch fails.

### 1.1 One-line pitch

> **The definitive QA and automated stabilization platform for AI coding agents.**

### 1.2 DiffVerify (Anti-Reward Hacking Innovation)

*This is Pare's core technical differentiator.*

When standard agents try to fix a failing test, they often succumb to "reward hacking" (e.g., fixing a red test by silently deleting the `assert`, or mocking out the actual business logic). 

Pare defeats this via **DiffVerify (Bidirectional Test Validation)**, marketed as *The Crucible*. To prevent a single LLM persona from writing "softball" tests to pass its own broken code, we enforce **Prompt Isolation**: test generation and code generation are strictly separated into two independent LLM calls with adversarial system prompts. Before any patch or automated repair is accepted, it must pass both directions:
1. **Backward Pass (Anti-cheat):** The *newly generated tests* successfully **fail** on the *old* (buggy) branch. (Proves the test catches the bug, rather than being a tautology).
2. **Forward Pass (Regression check):** The *old tests* successfully **pass** on the *new* (fixed) branch. (Proves no existing functionality was broken).

This guarantees the agent actually fixed the underlying logic, providing a strong empirical guardrail against reward hacking, under the assumption that existing tests reflect the intended specification.

### 1.3 The Core Product Loop: "Evaluate -> Audit -> Repair"

**1. The Evaluator (`pare eval`) - The Patch Inspector**
Batch-evaluates agent-generated patches against existing test suites and operational boundaries.

**2. The Auditor (Chaos Injection)**
Validates whether agent-generated patches can withstand edge-case inputs. 
* **Zero-Test Fallback (The Ephemeral Reproducer):** If the target codebase has no tests, the Auditor writes a standalone `pare_reproduce_bug.py` script that fuzzes the AI's new code until an exception is thrown, establishing a strict baseline test for future repairs. Applicable to crash-based issues in test-less codebases; not suitable for behavior-specification bugs that require explicit assertions.

**3. The Multiverse Repair Engine (`pare repair --universes 3`)**
For a failing patch generated by an agent, Pare doesn't try one fix and give up. It spawns **isolated parallel universes** using local `git worktree` and distinct personas concurrently to find the first verified working solution:
* **Universe A (Log_Hunter):** Traces runtime state.
* **Universe B (TDD_Purist):** Iterates strictly on failing tests.
* **Universe C (AST_Surgeon):** Refactors structural bottlenecks.

### 2.1 The Ghost Sandbox (Track Phase & Pit Stop)
True concurrency for full integration tests often fails due to hard requirements on shared resources (ports, SQLite locks, temp files). Pare solves this via **Partial Concurrency with Serialized Verification**:
* **1 Docker Container + 3 Worktrees**: Provides Python environment consistency (Docker) while allowing concurrent filesystem mutations (git worktree).
* **Track Phase (Parallel Thinking)**: Universes A, B, and C concurrently read code, analyze logs, and generate patches in their isolated worktrees.
* **Pit Stop Phase (Serial Verification)**: When a Universe generates a patch, it enters a `PitStopQueue`. Test execution is strictly serialized to avoid DB/port conflicts. During this phase, **Test Files Are Physically Locked**: a phase flag restricts `file_edit` and `file_create` tools via configurable path filters (e.g., `tests/`, `test_*.py`). Any attempt to modify tests during verification returns a hard `BLOCKED` error signal to the trajectory, completely shutting down "Validation Retreat" reward hacking. The first agent to clear the Pit Stop wins.

### 2.2 Heterogeneous Strategy Deck (Personas + Models)
Standard agents have one system prompt and rely on single-model consensus. Pare forces **Model Diversity** to repair broken patches:
* **Universe A (Claude 3.5 Sonnet)** + `TDD_Purist` (High context retention for architecture).
* **Universe B (GPT-4o)** + `Log_Hunter` (Fast tool execution for deep tracing).
* **Universe C (DeepSeek Coder)** + `AST_Surgeon` (Open-weight, raw code generation).

### 2.3 Cross-Universe Communication: The "Eureka" Broadcast
Complete isolation burns tokens (universes failing down the same dead end), but real-time syncing causes "Context Poisoning".
**The Solution (Asynchronous Blackboard with Rule-Based Gating):**
* Pare functions as a "Swarm sharing failure experiences". We provide a single read-write tool: `broadcast_eureka(type, error_trace)`.
* To prevent context poisoning and latency bloat, we strictly avoid LLM-based gating. Instead, we use **Rule-Based Gating**:
  - **Type Limiting**: Universes may only broadcast system-level blockers (`IMPORT_ERROR`, `MISSING_DEPENDENCY`, `ENV_BROKEN`). Hypothesis-level failures (`TEST_FAILURE`, `LOGIC_ERROR`) are dropped instantly so they don't mislead others.
  - **Rate Limiting**: Max 1 broadcast per N turns per universe.
* If Universe B crashes due to an obscure API migration missing a dependency, it broadcasts structured telemetry: `broadcast_eureka(type="MISSING_DEPENDENCY", error_trace="ModuleNotFoundError...")`.
* The Orchestrator intercepts this and injects a "Subspace Transmission" into Universes A and C as a normal `User` Observation, ensuring zero-latency pruning of dead-ends without hallucination contagion.

## 4. Value Proposition

As AI coding agents (like SWE-agent and OpenHands) proliferate, the volume of generated code will far exceed human capacity to review it. The bottleneck is no longer code generation, but code verification.

Pare solves this by providing a verifiable, un-hackable automated QA layer. By utilizing The Crucible to eliminate reward hacking and the Multiverse to autonomously repair flawed patches, Pare establishes the necessary trust to allow AI agents to operate at true scale.
